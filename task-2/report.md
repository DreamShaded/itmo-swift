## 4. Эксперименты с изменением запросов (Часть 3) — 8 шт.

Для каждого: исходный → изменённый → ответ → изменение состояния → баг/безопасность → исправление.

### E1. IDOR: лайк чужого кота по `catId`
- **Исходный:** лайк своего кота. **Изменённый:** `POST /api/likes/cats/18/likes {"like":true}` (кот `id=18`, мной не создан).
- **Ответ:** `200`, `likes 19 → 20`. **Состояние:** изменилось на сервере.
- **Безопасность:** да — Broken Object Level Authorization (IDOR). Любой объект меняется по id без проверки прав.
- **Исправление:** привязка действия к аутентифицированному пользователю + проверка прав на объект.

### E2. Накрутка: повтор одного и того же лайка
- **Изменённый:** `POST /api/likes/cats/48/likes {"like":true}` ×4.
- **Ответ:** `likes 1 → 2 → 3 → 4`. **Состояние:** счётчик растёт каждый раз.
- **Безопасность:** да — отсутствие идемпотентности и учёта «один пользователь = один голос».
- **Исправление:** хранить факт голоса на пользователя (уникальный ключ пользователь+кот), повтор = ноуоп/тогл; рейтлимит.

### E3. Правка описания чужого кота
- **Исходный:** описание `id=5` = `Запись для ручного редактирования"""`. **Изменённый:** `POST /api/core/cats/save-description {"catId":5,"catDescription":"TAMPERED-BY-QA"}`.
- **Ответ:** `200`; повторный GET подтвердил персист на сервере. *(Восстановлено в исходное.)*
- **Безопасность:** да — изменение чужих данных без авторизации.
- **Исправление:** серверная проверка владельца/прав записи.

### E4. Дубликат имени
- **Изменённый:** `POST /api/core/cats/add` с именем `Леопольд` (уже существует, `id=30`).
- **Ответ:** `200` `{"cats":[{"errorDescription":"Такое значение уже существует"}]}`.
- **Баг:** дубликаты отклоняются (хорошо), но статус `200` вместо `409 Conflict` (BUG-07).
- **Исправление:** возвращать `409` с машиночитаемым кодом ошибки.

### E5. Невалидные значения
- `gender:"attack-helicopter"` → `200`, кот создан с `gender:"unisex"` — **невалидный enum молча заменён** (SEC-08).
- `name:"12345!!!<script>"` → `200` `{"errorDescription":"Цифры не принимаются!"}` — валидация имени **есть** на сервере, но статус `200` (BUG-03).
- `catDescription` из 5000 символов → `200`, принято целиком — **нет лимита длины** (BUG-05).
- `search {"name":"123"}` → `200` `{"groups":[]}`.
- **Исправление:** строгая enum-валидация → `400`; лимиты длины; единый формат ошибок c кодами.

### E6. Вызов теневого web-API (нет в приложении)
- `POST /cats/48/like` → `200 OK` (легаси-эндпоинт работает, иной формат ответа).
- `DELETE /cats/48/dislike` → **`500`** с **полной ошибкой PostgreSQL**: `constraint "dislikes_positive"`, `table "cats"`, `code 23514`, `detail: Failing row contains (48, Котозонд, …, 5, -1)`, `file execMain.c`, `line 2089` (SEC-05, BUG-04).
- `GET /cats/dislikes-rating` → `200` (работает).
- **Безопасность:** да — широкая теневая поверхность + утечка схемы БД.
- **Исправление:** убрать/закрыть легаси-API; не отдавать stacktrace/SQL клиенту.

### E7. Несуществующий `catId`
- `GET /api/core/cats/get-by-id?id=999999` → `404` `{"message":"cat not found"}`.
- `POST /api/likes/cats/999999/likes` → `404`. `save-description catId:999999` → `404`.
- **Поведение корректное** (единственное место с правильными статусами). Замечание: формат ошибок (Boom) отличается от других мест (BUG-06).

### E8. Загрузка файлов
| Файл         | Содержимое   | Расширение | MIME              | Ответ                            |
| ------------ | ------------ | ---------- | ----------------- | -------------------------------- |
| good.png     | реальный PNG | .png       | image/png         | `200 {"fileUrl":…}`              |
| notimage.png | **текст**    | .png       | image/png         | **`200` — принят как картинка!** |
| evil.php     | `<?php …`    | .php       | application/x-php | `500` (краш, не 4xx)             |
| image.txt    | реальный PNG | .txt       | text/plain        | `500` (краш)                     |

- **Состояние:** текстовый файл сохранён и **отдаётся как `image/png`** с `:3001/photos/…` и `:8080/photos/…` (Content-Type навязан по расширению).
- **Безопасность:** да — нет проверки сигнатуры/содержимого; решение по расширению; отказ = `500` (SEC-06, BUG-04).
- **Исправление:** проверять magic-bytes, белый список MIME по содержимому, перекодировать изображение, отдавать `Content-Type` по факту + `X-Content-Type-Options: nosniff`.

## 5. Эксперименты с изменением ответов (Часть 4) — 4 шт.

Подмена ответов через Charles **Map Local / Rewrite** (или Proxyman). Ключевой принцип проверен экспериментально: **изменение ответа меняет только то, что видит клиент; источник истины — сервер.** После любой подмены повторный «чистый» GET (без правила) возвращает реальные данные → правка на сервере не сохраняется.

| #   | Что подменяем (Rewrite в ответе)                        | Что видит приложение                | Сохранилось на сервере? | Проверка                            |
| --- | ------------------------------------------------------- | ----------------------------------- | ----------------------- | ----------------------------------- |
| R1  | `name`/`description` кота в ответе `search`/`get-by-id` | Фейковое имя/описание в карточке    | **Нет**                 | повторный GET → реальное имя        |
| R2  | `likes`/`dislikes` (напр. 42 → 9999)                    | Накрученный счётчик в UI            | **Нет**                 | GET с сервера → исходное число      |
| R3  | весь `groups[]` в `search` (добавить «своих» котов)     | Несуществующие коты в списке        | **Нет**                 | их нет в `/cats/all`                |
| R4  | `images[]`/`fileUrl` → внешний URL картинки             | Приложение грузит чужое изображение | **Нет**                 | список фото на сервере не изменился |

**Граница клиент/сервер:** правки **ответов** (Часть 4) — всегда только на клиенте. А вот правки **запросов** (Часть 3: E2 лайки, E3 описание, E8 файл) **реально сохраняются на сервере**, потому что сервер слепо доверяет вводу клиента. Именно это и есть главный урок (см. §8).

*(Скриншоты «до/после» в самом приложении не приложены: работа велась через прямые HTTP-запросы без управления UI телефона. Поведение клиент-vs-сервер доказано через повторные GET-запросы — см. evidence/log.md.)*

## 6. Находки по безопасности (Часть 5) — 8 шт.

### SEC-01 — Полное отсутствие аутентификации и авторизации
- **Серьёзность:** Критический
- **Предусловия:** сетевой доступ к `:3001`.
- **Шаги:** любой запрос к любому эндпоинту без токена/сессии/заголовка.
- **Факт:** все методы (включая запись и удаление) доступны анонимно; «вход» по имени+телефону — чисто клиентский, серверной сессии нет (нет `/login` в спеке и DEX).
- **Ожидается:** аутентификация (сессия/токен) для мутаций; защищённые операции требуют авторизации.
- **Доказательство:** отсутствие `securitySchemes` в спеке; все E1–E8 без авторизации.
- **Рекомендация:** ввести аутентификацию и серверные сессии; закрыть мутации для анонимов.

### SEC-02 — IDOR / Broken Object Level Authorization
- **Серьёзность:** Высокий
- **Шаги:** `POST /api/likes/cats/18/likes`, `POST /api/core/cats/save-description {"catId":5,…}` для чужих котов.
- **Факт:** изменение лайков и описания любых объектов по id; `likes 19→20`, описание `id=5` перезаписано.
- **Ожидается:** действие разрешено только владельцу/авторизованной роли.
- **Доказательство:** E1, E3 (log.md).
- **Рекомендация:** проверка прав на объект на сервере.

### SEC-03 — Неаутентифицированное удаление данных
- **Серьёзность:** Высокий
- **Шаги:** `DELETE /cats/{catId}/remove`.
- **Факт:** кот удаляется без авторизации (мои `id=48,50` исчезли; в проде кот `id=30` имеет описание «Леопольд вернулся. Больше не удаляйте» — след чужого удаления).
- **Ожидается:** удаление требует прав; недоступно анонимно.
- **Доказательство:** E6/очистка (log.md).
- **Рекомендация:** авторизация + soft-delete/аудит.

### SEC-04 — Накрутка голосов (нет идемпотентности/рейтлимита)
- **Серьёзность:** Средний
- **Шаги:** повтор `POST /api/likes/cats/{id}/likes {"like":true}`.
- **Факт:** счётчик растёт неограниченно (1→2→3→4 за 4 запроса); рейтинг легко накручивается.
- **Ожидается:** один голос на пользователя; тогл; ограничение частоты.
- **Доказательство:** E2.
- **Рекомендация:** учёт голосов на пользователя, рейтлимит.

### SEC-05 — Раскрытие внутренней информации в ошибках
- **Серьёзность:** Высокий
- **Шаги:** `DELETE /cats/48/dislike` (увод счётчика < 0).
- **Факт:** `500` с полным дампом PostgreSQL: имя БД, таблица `cats`, имена столбцов и значения строки, имя ограничения `dislikes_positive`, SQLSTATE `23514`, `execMain.c:2089`.
- **Ожидается:** обобщённое сообщение `500` без внутренних деталей.
- **Доказательство:** E6 (log.md, строки с `Failing row contains`).
- **Рекомендация:** глобальный error-handler, скрывающий stacktrace/SQL; логировать детали только на сервере.

### SEC-06 — Загрузка файлов без проверки содержимого + навязанный Content-Type
- **Серьёзность:** Высокий
- **Шаги:** загрузить текстовый файл с именем `*.png` и `Content-Type: image/png`.
- **Факт:** `200`; файл сохранён и отдаётся из `/photos/*` с `Content-Type: image/png` (определяется по расширению, не по содержимому). Открывает путь к сохранённому контенту, управляемому атакующим (потенциальный stored-XSS при `.svg`/`.html`).
- **Ожидается:** проверка magic-bytes/реального MIME, перекодирование изображений, корректный Content-Type, `nosniff`.
- **Доказательство:** E8; `curl /photos/image-…png` → отдаётся текст как image/png.
- **Рекомендация:** валидация содержимого, белый список форматов, изоляция хранилища, `X-Content-Type-Options: nosniff`.

### SEC-07 — Дублирующая «теневая» поверхность API
- **Серьёзность:** Средний
- **Шаги:** вызвать `/cats/*` (легаси), которых нет в приложении.
- **Факт:** параллельный API с иным поведением и форматами ответов (`OK` текстом, иные ошибки), включая опасный `DELETE /cats/{id}/remove`. Расширяет поверхность атаки и расходится с `/api/*`.
- **Ожидается:** единый поддерживаемый API; легаси закрыт.
- **Доказательство:** §2.2, E6.
- **Рекомендация:** удалить/закрыть легаси-маршруты; единые контракты и валидация.

### SEC-08 — Молчаливая подмена невалидного enum (слабая валидация ввода)
- **Серьёзность:** Средний
- **Шаги:** `POST /api/core/cats/add` с `gender:"attack-helicopter"`.
- **Факт:** `200`, кот создан с `gender:"unisex"` — сервер проглотил мусорное значение вместо ошибки.
- **Ожидается:** `400 Bad Request` на значение вне enum.
- **Доказательство:** E5 (создан `id=50`).
- **Рекомендация:** строгая валидация запросов по схеме (белый список enum), отказ при нарушении.

> Дополнительно (инфраструктура, Средний): только **HTTP** (открытый текст), отсутствуют заголовки безопасности (`HSTS`, `CSP`, `X-Content-Type-Options`, `X-Frame-Options`), утечка `X-Powered-By: Express`, нет рейтлимита. Усиливает SEC-04/06 и допускает MITM.

## 8. Вывод: почему нельзя доверять мобильному клиенту

Эксперименты прямо показали разрыв между ограничениями клиента и проверками сервера:

1. **Клиентские правила обходятся.** Валидация имени (`/cats/validation`) и enum `gender` в приложении не защищают сервер: запросы можно слать напрямую, минуя UI. Сервер либо принимал мусор (`gender` → SEC-08), либо защищал частично и с неверными статусами (BUG-02/03).
2. **Подмена ответов ≠ изменение данных.** Charles Rewrite (Часть 4) меняет только то, что рисует клиент — сервер остаётся источником истины. Опасны не подменённые ответы, а **подменённые запросы**: лайки, описание, загрузка файла (Часть 3) реально сохранились, потому что сервер доверяет вводу.
3. **Нет идентичности — нет авторизации.** Отсутствие аутентификации (SEC-01) делает невозможными проверки владения, отсюда IDOR (SEC-02), анонимное удаление (SEC-03) и накрутка (SEC-04).

**Что должно проверяться на сервере:**
- аутентификация и серверные сессии; авторизация на уровне объекта (владелец/роль) для всех мутаций;
- строгая валидация по схеме (enum, длины, типы) с `400` при нарушении — независимо от клиента;
- идемпотентность и рейтлимит для голосов и загрузок;
- валидация файлов по содержимому, безопасная отдача (`nosniff`, корректный Content-Type);
- единый формат ошибок без раскрытия stacktrace/SQL; корректные HTTP-статусы;
- закрытие легаси/теневого API; HTTPS + заголовки безопасности.

Клиент — лишь удобный интерфейс. Любое правило, не продублированное проверкой на сервере, отсутствует с точки зрения безопасности.
